# 30\_属性生成与随机性验证

宝石属性并不是由运营方手动输入。挖矿时，挖矿合约会基于随机值决定 `Weight`、`Color`、`Clarity` 和 `Cut`。

关键在于，随机生成方式必须可验证。如果运营方可以任意生成高等级 NFT，CryptoStone 的去中心化性和稀缺性将被破坏。

因此，CryptoStone 的随机生成方式遵循以下原则。

| 原则              | 说明                               |
| --------------- | -------------------------------- |
| 不可提前预测          | 运营方和用户都不能提前知道结果。                 |
| 不能选择结果          | 运营方不能只选择有利结果或拒绝不利结果。             |
| 防止重试            | 随机请求之后、结果确定之前，用户不得取消 claim 或重试。  |
| 禁止 Admin Reroll | 运营方不得对特定结果进行重新抽取或替换。             |
| 公开验证            | 结果生成过程必须在链上或以公开可验证方式进行。          |
| 固定概率表           | 属性概率表必须在部署前公开，并在 finalize 后不可更改。 |
| 发行后不可变          | 发行后属性不得更改。                       |

初期实现中，可以优先考虑使用可验证的外部随机基础设施。也可以采用 Commit-then-Reveal 等方式，将随机请求时间与结果确定时间分离。无论采用何种方式，用户都不得在结果确定前取消 claim 或重试，运营方也不得选择或重新抽取特定结果。

随机结果被请求后，用户不得因结果不利而取消 claim 或重试。运营方也不得为特定用户分配有利结果，或选择能够产生特定属性组合的 `seed`。这是为了确保稀有宝石由可验证随机结构产生，而不是由运营方裁量产生的核心条件。

发行后，Gem NFT 的核心属性必须被 `freeze`，`Weight`、`Color`、`Clarity`、`Cut`、`stoneType`、`minedAt`、`minedFromPool` 等核心 metadata 不得由运营方更改。

初期可以使用实务上可验证的随机基础设施，长期则可以发展为由 CryptoStone 网络自身负责随机生成与验证的结构。
