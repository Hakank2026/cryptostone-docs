# 30\_属性生成とランダム性の検証

宝石の属性は、運営者が手動で入力するものではない。採掘時点で、マイニングコントラクトがランダム値を基に `Weight`、`Color`、`Clarity`、`Cut` を決定する。

重要なのは、ランダム生成方式が検証可能でなければならないという点である。もし運営者が恣意的に高等級のNFTを生成できるなら、CryptoStoneの分散性と希少性は損なわれる。

したがって、CryptoStoneのランダム生成方式は次の原則に従う。

| 原則             | 説明                                               |
| -------------- | ------------------------------------------------ |
| 事前予測不可         | 運営者とユーザーの双方が結果を事前に知ることはできない。                     |
| 結果選択不可         | 運営者は有利な結果のみを選択したり、不利な結果を拒否したりできない。               |
| 再試行防止          | ランダムリクエスト後、結果確定前にユーザーがclaimをキャンセルまたは再試行できてはならない。 |
| Admin Reroll禁止 | 運営者は特定結果を再抽選または置換できてはならない。                       |
| 公開検証           | 結果生成プロセスはオンチェーンまたは公開検証可能な方式でなければならない。            |
| 確率表固定          | 属性確率表はデプロイ前に公開され、finalize後に変更できてはならない。           |
| 発行後不変          | 発行後、属性は変更できてはならない。                               |

初期実装では、検証可能な外部ランダムインフラを優先的に検討できる。または、Commit-then-Reveal構造のような方式で、ランダムリクエスト時点と結果確定時点を分離することもできる。いずれの方式を使用する場合でも、ユーザーは結果確定前にclaimをキャンセルまたは再試行できてはならず、運営者も特定結果を選択または再抽選できてはならない。

ランダム結果がリクエストされた後、ユーザーは結果が不利であるという理由でclaimをキャンセルまたは再試行できてはならない。運営者もまた、特定ユーザーに有利な結果を割り当てたり、特定属性組み合わせが出るようにseedを選択したりできてはならない。これは、希少宝石の生成が運営者の裁量ではなく、検証可能なランダム構造によって決定されるようにするための核心条件である。

発行後、Gem NFTの核心属性はfreezeされなければならず、`Weight`、`Color`、`Clarity`、`Cut`、`stoneType`、`minedAt`、`minedFromPool` といった核心metadataは、運営者によって変更されてはならない。

初期には実務的に検証可能なランダムインフラを使用しつつ、長期的にはCryptoStoneネットワーク自体がランダム生成と検証を担う構造へ進化できる。
